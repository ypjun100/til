# 도커 스웜

- 도커 엔진에 내장된 컨테이너 오케스트레이션 도구로, 여러 도커 서버를 하나의 클러스터로 묶어 관리함
	- 오케스트레이션 도구란 컨테이너를 분산 배포, 관리할 수 있게 하는 도구임
- **주요 기능**
	- 로드 밸런싱 : 내부 DNS를 통해 서비스에 자동으로 DNS 부여 및 요청을 분산 처리
	- 오토 스케일링 : 컨테이너 수를 유동적으로 조절 가능
	- 멀티 호스트 네트워킹 : 오버레이 네트워크로 여러 노드에 걸친 컨테이너 통신 지원
	- 무중단 업데이트/롤백 : 서비스 설정 변경 시 수동 재시작 없이 자동으로 반영

### 도커 스웜 모드의 구조

- 스웜 모드는 매니저 노드와 워커 노드로 구성되며, 워커는 없어도 되지만 매니저 노드는 1개 이상 있어야 함
	- **매니저 노드** : 클러스터 상태 관리, 서비스 스케쥴링, 워커 노드 관리 및 워커 노드 역할도 겸함
	- **워커 노드** : 실제 컨테이너가 생성 및 운영되는 도커 서버
- 매니저 노드의 과반수가 장애가 발생하는 경우 매니저 노드가 복구될 때까지 클러스터 운영을 중단함
	- 짝수 개의 매니저로 구성한 클러스터는 네트워크 파티셔닝 현상이 발생하면 운영이 중단될 수 있음
	- 홀수 개로 구성하면 과반수 이상이 유지되는 쿼럼 매니저에서 운영을 계속할 수 있음

> [!tip]
> 매니저 노드는 일반적인 매니저 역할을 하는 노드와 리더 역할을 하는 노드로 나뉜다. 리더 매니저의 서버에 장애가 생기면 새로운 리더를 선출해야 하는데, 이때 Raft Consensus 알고리즘을 사용한다. ([Raft Consensus 알고리즘 설명](https://seongjin.me/raft-consensus-algorithm/))

> [!Question] 워커 노드 대신 매니저 노드로만 클러스터를 구성하면 안 되는가?
> 만약 노드가 3개가 있는 클러스터 환경에서 모든 노드를 매니저 노드로서 구성하면, 매니저 리더 1개가 죽어도 나머지 2개가 쿼럼(과반수)를 충족하여 자동으로 새 리더를 선출하게 되며, 클러스터 무중단 유지가 가능하다. 하지만, 매니저가 많아질수록 Raft 알고리즘으로 상태를 동기화하는 오버헤드가 증가하고, 매니저는 기본적으로 클러스 관리 작업에 리소스를 사용하므로 워커 노드보다 컨테이너 처리 효율이 낮다. 따라서, 실제 운영 환경에서는 매니저 노드에 컨테이너 배포는 제한하는 것이 일반적이다.

### 도커 스웜 모드 사용 예시

1. **트래픽이 많아 서버 1대로 부족할 때** : 여러 노드에 컨테이너를 분산하고 로드밸런싱으로 요청을 나눔
2. **무중단 서비스 필요 시** : 특정 노드가 죽어도 다른 노드에서 컨테이너가 자동으로 재생성되어 서비스 유지
3. **배포 중 다운타임이 없어야 할 때** : 롤링 업데이트로 컨테이너를 순차 교체하여 무중단 배포가 가능
4. **컨테이너 수를 조절해야 할 때** : 트래픽에 따라 레플리카 수를 늘리거나 줄이는 오토 스케일링 가능
5. **마이크로서비스 아키텍처를 운영할 때** : 여러 서비스를 클러스터 위에서 통합 관리
