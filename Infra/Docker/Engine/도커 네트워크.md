# 도커 네트워크

| 구분    | docker0                 | veth                 |
| ----- | ----------------------- | -------------------- |
| 종류    | 가상 브리지                  | 가상 이더넷               |
| 생성 시점 | Docker 설치 시 자동 생성       | 컨테이너 생성 시마다 생성       |
| 개수    | 기본 1개(커스텀 네트워크 추가 시 증가) | 컨테이너당 1개             |
| 역할    | 여러 veth를 하나로 묶어 패킷 중재   | 컨테이너 <-> 호스트 간 패킷 전달 |
- 도커 컨테이너에서 외부 통신을 하기 위해서 도커 내부적으로 네트워크 인터페이스를 구성하게 됨
	- `docker0`은 여러 컨테이너를 하나로 묶는 허브, `veth`는 컨테이너와 호스트를 1:1로 연결하는 터널
- 컨테이너를 생성하면 `docker0` 브리지를 통해 외부 통신을 구성할 수 있지만, 다른 드라이버도 사용 가능함
	- 도커에서 제공하는 드라이버로 `bridge`, `host`, `none`, `container` 등이 있음

### 브리지 네트워크

```shell
# 새로운 브리지 네트워크 생성
docker network create --driver bridge mybridge
```

- `docker0`이 아닌 사용자 정의 브리지를 새로 생성해 각 컨테이너에 연결하는 네트워크 구조
- `docker run` 명령어에 `--net` 옵션을 설정하면 컨테이너가 해당 네트워크를 사용하도록 설정 가능함
	- 새로운 브리지 네트워크로 컨테이너를 생성하면 해당 컨테이너의 IP 대역은 새로운 대역으로 할당됨
- 컨테이너에 설정된 네트워크는 `docker network disconnect/connect`로 유동적인 연결/해제가 가능
	- `bridge`, `overlay` 네트워크 만 사용할 수 있으며, `host`, `none`에서는 사용할 수 없음

> [!TIP]
> 기본 `docker0` 브리지는 컨테이너 이름으로 통신이 불가능하지만, 사용자 정의 브리지는 컨테이너 이름으로 DNS 통신이 가능하다. 따라서 실무에서는 `docker0`을 직접 쓰는 것보다 사용자 정의 브리지를 만들어 사용하는 것이 권장된다.

> [!Question]
> `docker network disconnect/connect` 명령어를 `bridge`, `overlay` 네트워크에서만 사용 가능한 이유는, 해당 네트워크는 도커 내부적으로 가상 네트워크 터널을 생성하고 삭제하는 방식으로 동작하기 때문이다. 반면 `host` 네트워크는 컨테이너가 OS 네트워크 스택을 사용하여 호스트의 네트워크 네임스페이스를 직접 공유하므로, "연결"이라는 개념이 없고, `none` 네트워크는 아예 네트워크 인터페이스가 없는 격리 상태이다. 

### 호스트 네트워크

```shell
# host 네트워크를 사용하여 컨테이너 생성
docker run -it --name container1 --net host ubuntu
```

- 도커를 설치하면 자동으로 `bridge`, `host`, `none` 네트워크가 각각 하나씩 생성됨
	- 해당 네트워크들은 기본 네트워크로, `docker network rm` 명령어로 제거할 수 없음
- 호스트 드라이버는 브리지 드라이버와 달리 별도로 생성할 필요없이 기존의 host라는 네트워크를 사용함
- 호스트 네트워크를 설정하면 컨테이너 내에서도 호스트 머신의 네트워크 설정 정보가 그대로 출력됨
- 또한, 호스트 네트워크로 설정하면 컨테이너 내부의 애플리케이션을 별도 포트 포워딩 없이 서비스 가능함

### 논 네트워크

```shell
# none 네트워크를 사용하여 컨테이너 생성
docker run -it --name container2 --net none ubuntu
```

- `none`은 말 그대로 아무런 네트워크를 쓰지 않는 것을 의미하며, 컨테이너 생성시 외부와의 연결이 단절됨
- 컨테이너 내부에 설정된 네트워크 인터페이스도 로컬호스트를 나타내는 `lo` 외에는 존재하지 않음

> [!TIP]
> **`none` 네트워크를 사용하는 경우**
> - **민감 데이터 처리** : 네트워크 통신이 필요없는 민감한 데이터를 처리하는 경우
> - **네트워크 불필요 순수 연산 작업** : 머신러닝 모델 학습, 암호화 처리와 같이 외부 통신이 필요없는 경우

### 컨테이너 네트워크

```shell
# 첫 번째 컨테이너 생성
docker run -itd --name network_container_1 ubuntu

# 두 번째 컨테이너 생성
docker run -itd --name network_container2 \
--net container:network_container_1 ubuntu
```

- 새 컨테이너가 기존의 실행 중인 컨테이너의 네트워크 네임스페이스를 그대로 공유하는 방식임
- 두 컨테이너가 동일한 네트워크 네임스페이스를 사용하며, IP, MAC 주소를 서로 공유하게 됨
- 프로세스는 각각 독립적으로 분리되지만, IP를 공유하기 때문에 두 컨테이너가 같은 포트를 열면 충돌함

> [!TIP]
> **`container` 네트워크를 사용하는 경우**
> - **사이드카 패턴** : 메인 앱 컨테이너 옆에 로그 수집기, 모니터링 에이전트 등을 붙이는 경우
> - **디버깅** : 실행 중인 컨테이너의 네트워크 상태를 들여다보는 디버깅용 컨테이너를 붙이는 경우
