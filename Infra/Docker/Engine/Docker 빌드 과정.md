# Docker 빌드 과정

```
=> [2/6] WORKDIR /app
=> [3/6] RUN npm install
=> [4/6] ADD . /app/
```

- `[2/6]`처럼 표시된 부분은 `Dockerfile`에  기록된 명령어의 순서에 해당되며, 차례대로 실행됨
- `Dockerfile`의 명령어가 실행될 때마다 컨테이너를 생성한 뒤 해당 명령어를 실행하고 이미지로 커밋함
	- 도커는 파일 시스템에 실제 변경 사항이 발생할 때만(`RUN`, `COPY`, `ADD`) 새로운 레이어를 생성함
	- `CMD`, `LABEL`, `ENV`, `EXPOSE` 등은 이미지 사이즈에 영향을 주지 않는 임시 레이어로 처리됨
	- `RUN`을 줄일수록 이미지 크기가 작아지기에,  명령어를 `&&`로 묶는 게 일반적인 최적화 방법임
		```Dockerfile
		RUN apt-get update && apt-get install -y curl
		```

> [!TIP]
> `Dockerfile` 빌드 중 생성되는 임시 컨테이너는 도커가 아닌 buildkit이라는 별도의 프로세스에서 제어되므로 `docker ps` 명령어로는 확인할 수 없다.

> [!Question]
> 도커 이미지 레이어는 읽기 전용이라 한번 생성되면 수정이 불가능하다. 이후 레이어에서 파일을 삭제해도 이전 레이어의 파일은 그대로 남고, 삭제 표시만 새 레이어에 추가된다. 따라서, `&&`로 하나의 `RUN`에 체이닝하면 해당 레이어의 최종 결과물에 임시 파일이 포함되지 않아 용량이 줄어든다. **즉, 레이어 수를 줄이는 목적은 중간 과정의 임시 파일이 레이어에 박제되는 것을 방지하기 위함이다.**

### 캐시를 이용한 이미지 빌드

- 이미지 빌드를 마치고 난 뒤 다시 동일 빌드를 진행하면, 이전 이미지 빌드에서 생성된 캐시를 사용함
- 이미지 캐시를 사용함으로써 불필요하게 동일한 명령어에 대한 재빌드를 할 필요 없이 캐시를 이용하게됨
- 만약 이미지 빌드 시에 캐시를 이용하고 싶지 않다면, `build` 명령에 `--no-cahce` 옵션을 추가함
	- ex) `docker build --no-cache -t mybuild:0.0 .`
	- `RUN git clone something`과 같이 멱등성이 부합되지 않는 명령은 캐시를 사용하면 안 됨
