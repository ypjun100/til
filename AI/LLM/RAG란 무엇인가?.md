# RAG란 무엇인가?

### 대규모 언어 모델의 한계

- **환각 발생**: 모델이 사실과 다른 내용을 그럴듯하게 생성해도 이를 스스로 검증하지 못함
- **정적 지식의 한계**: 모델이 학습된 시점 이후의 정보를 반영하지 못함
- **맞춤형 데이터 부족**: 기업 내부 문서는 학습데이터로 포함되기 어렵기 때문에 현업에서 사용할 수 없음
- **출처 추적 어려움**: 모델이 내놓은 답변은 근거를 명확히 제시하지 못함
- **업데이트 비용 문제**: 새로운 지식을 반영하려면 모델 전체를 재학습해야 함

### 환각은 왜 발생하는 것인가?

- LLM(대규모 언어 모델)에서 흔히 발생되는 환각은 크게 두 가지의 유형이 있음
    - 내재적 환각: 학습한 정보를 잘못 조합하거나 문맥을 왜곡할 때 발생
    - 외재적 환각: 지식 범위를 넘어선 질문에 대해 근거 없는 답변을 내놓을 때 발생
- LLM은 불완전한 확률 분포를 바탕으로 다음 단어를 예측하기 때문에 불확실성이 크면 환각 발생 확률이 높음
- 특히, 언어 모델이 다음 토큰을 어떻게 정하는지 정하는 전략에 따라 환각 발생 확률이 높아짐
    - Top-k 샘플링: 모델이 예측한 토큰들 중 확률이 높은 K개만 남기고 그 안에서 하나를 뽑는 방식
    - Nucleus 샘플링: 토큰을 내림차순으로 누적 확률이 p 이상이 되는 최소 집합에서 하나를 뽑는 방식

### LLM vs RAG

- **지식 소스**: LLM은 학습 시점에 내재된 지식에만 의존하지만, RAG는 사전 지식, 외부 문서를 동시에 활용
- **맥락 처리 능력**: LLM은 입력 토큰 윈도우 범위에서만 맥락 고려, RAG는 검색된 외부 문맥을 주입함
- **확장성**: LLM에 새 지식을 반영하려면 재훈련이 필요하지만, RAG는 지식베이스만 갱신하여 최신 정보 반영
- **응답 품질**: LLM은 최신 사실이나 특수 도메인에는 취약, RAG는 검색 기반 보강으로 정확성 확보 가능

### RAG 프로세스

1. **(사전) 데이터 인덱싱**
    - RAG가 검색할 수 있는 지식베이스를 미리 만들어두는 단계로 여러 외부 데이터를 정규화하는 단계
    - 특히, 긴 문서를 작은 조각으로 나누는 청킹 작업 이후의 청크는 벡터로 변환되어 저장됨
        - 청킹 작업에서 청크가 너무 크면 검색은 넓게 되지만 불필요한 정보가 섞이고, 너무 작으면 문맥이 끊겨 검색 품질이 떨어질 수 있음
    - 이 단계는 검색 품질의 상한선을 정하기 때문에 중요한 단계임
2. **질의 전처리 및 임베딩**
    - 사용자가 질문을 입력하면 이를 검색 가능한 형태로 변환하는 단계
    - 전처리 단계에서는 오타 보정, 불필요한 표현 제거, 약어 확장, 언어 감지, 질의 재작성 등이 이뤄짐
    - 그 이후 사용자 질문도 문서 청크와 동일한 벡터 공간에 있는 임베딩 모델로 벡터화함
3. **문서 검색**
    - 질의 벡터와 벡터 DB 안의 문서 벡터를 비교해 가장 가까운 청크를 찾음
    - 보통 상위 `k`개를 가져오며, 이를 유사도가 높은 청크 10개의 후보라고 가정함
        - 유사도 계산에는 코사인 유사도, Dot Product, L2 Distance 방식이 있음
        - **코사인 유사도**: 벡터 간 각도를 기준으로 방향의 유사성을 평가하는 방식으로, 벡터 크기는 무시하고, 순수한 의미적 유사성에 집중하기 때문에 가장 널리 사용됨
        - **Dot Product(내적)**: 벡터의 크기와 방향을 모두 고려하는 방식으로, 정규화된 벡터에서는 코사인 유사도와 같은 순위를 제공하면서도 계산이 빠름
        - **L2 Distance(유클리스 거리)**: 두 벡터 간 절대적인 직선 거리를 평가하는 방식으로, 정확한 위치 기반 계산에는 적합하지만, 고차원 벡터에서는 계산 효율성이 떨어질 수 있음
        - 거리 계산 방식은 보통 임베딩 모델이 어떤 방식으로 학습되었는지에 따라 달라짐
4. **재순위화**
    - 벡터 유사도가 높다고 실제 질문에 정확한 답변이라는 보장은 없기에 선택적으로 re-ranker를 사용함
    - re-ranker는 질문과 문서를 한 쌍으로 넣어, 문서가 질문의 답변으로 유용한지를 평가함
    - 대게 검색 결과를 가져온 후에 re-ranker가 그 중 상위 N개만 최종 컨텍스트로 고르는 방식임
    - 이 단계는 정확도를 높이지만 비용과 지연 시간이 늘어난다는 단점이 존재함
5. **프롬프트 증강**
    - 검색된 문서 조각을 사용자의 질문과 결합해 LLM 입력을 만드는 단계임
    - 단순히 문서를 합치는 게 아닌, LLM이 근거를 잘 사용하도록 구조화하는 것이 중요함
6. **응답 생성 및 후처리**
    - LLM이 답변을 생성하는 데 일반 지식뿐만 아니라 증강된 프롬프트를 바탕으로 답변을 생성함
    - 특히, 출처 표기는 RAG의 신뢰성을 높이는 핵심 요소로 사용자는 답변의 출처를 추적할 수 있음

### 백터 임베딩과 백터 데이터베이스

- 임베딩은 단어·문장·문서를 다차원 공간의 벡터로 바꿔서, 의미적 유사성을 수학적으로 계산할 수 있도록 함
- RAG에서는 사용자 질의와 데이터를 같은 임베딩 공간에 투영한 뒤, 두 벡터 간 유사성을 계산함
- 이렇게 만들어진 벡터들을 저장하고 빠르게 검색하는 역할은 백터 데이터베이스가 담당하게 됨
- **백터 DB 예시**
    - **Pinecone**: 서버 설치, 샤딩, 확장 같은 인프라 부담을 줄인 완전 운영형 벡터 DB 서비스
    - **Milvus**: 성능과 확장성을 기반으로 대규모 검색과 인프라 제어력을 제공하는 오픈소스 벡터 DB
    - **ChromaDB**: 기본적인 기능을 제공하며, LLM/RAG 앱 개발자가 빠르게 쓰기 좋은 벡터 DB
    - **Qdrant**: 특정 데이터를 찾기 위한 필터링 중심의 벡터 DB
    - **Redis**: Redis Stack 위에서 벡터 검색을 제공하는 메모리 중심 DB

### 검색기(Retriever)

- 검색기는 사용자의 질의와 관련된 문서를 벡터 DB에서 빠르고 효율적으로 찾아내는 역할을 수행함
    - 검색 방식은 크게 희소(Sparse) 검색과 밀집(Dense) 검색으로 나눌 수 있음
    - **희소 검색(Sparse Retrieval)**: 전통 정보 검색 기법을 활용해 키워드 일치를 기반으로 문서를 찾는 방식으로, 특정 용어가 반드시 포함된 문서를 빠르게 찾아내는 데 유리하지만 동의어에는 취약함
    - **밀집 검색(Dense Retrieval)**: 임베딩 모델로 문서를 벡터로 변환하고 코사인 유사도나 내적으로 의미적 유사성을 평가하는 방식으로, 동의어를 찾아내는 데는 좋지만, 실시간 추론 비용이 큼
- Dense와 Sparse 검색은 서로 보완적인 특성을 가지고 있기에, 두 방식을 결합한 하이브리드 검색이 활용됨
    - 두 검색을 병렬로 수행한 뒤 적절한 방식으로 결합하고 이를 기반으로 최종 순위를 조정하는 방식
    - 정확한 팩트 검색이 중요하다면 Sparse에, 의미적 연관성이 중요하다면 Dense의 비중을 높임

> [!NOTE] Sparse/Dense의 의미론적 의미
> - Sparse는 ‘희박한’의 의미로, 문장의 벡터에서 대부분은 0이고, 특정 단어 부분에만 값이 들어가기에 차원이 수만 개가 될 수 있는데도 실제 값은 극히 일부분만 채워져, Sparse라고 부름
> - Dense는 ‘빽빽한’의 의미로, Sparse와 달리 대부분의 값들이 0이 아니고 값이 촘촘하게 채워져 있어서, Dense이라고 부름

### 문서 청킹 전략

- 청킹이 제대로 안 되면 개념이 여러 조각으로 흩어지거나, 다른 내용이 한 덩어리에 묶여 정확도가 떨어짐
- 따라서, 문서 유형, 데이터 특성을 고려해 전략을 짜는 게 중요하며, 아래는 대표적인 청킹 방법론임
- **고정 크기 청킹** : 일정 토큰 수나 문자 수 기준으로 균일하게 분할하는 가장 단순한 방식이지만, 문장이나 단락 경계를 무시해 의미가 끊길 수 있음
- **재귀적 문자 청킹** : 구두점이나 개행 문자를 우선적으로 고려해 분할하는 방식으로, 단순 고정 크기보다 문서 구조를 보존할 수 있음
- **의미적 청킹** : LLM이나 임베딩을 활용해 의미 단위로 잘라내는 정교한 방법으로, 복잡한 문서에 특히 효과적이지만, 계산 비용이 큼
- **구조 기반 청킹** : 문서 자체의 형식을 활용하는 빙식으로, 마크다운은 헤더 레벨, HTML은 태그 레벨을 기반으로 청킹

### RAG 성능 측정 방법

- **Recall@K**
	```
	Recall@K = 상위 K개 안에 포함된 관련 문서 수 / 전체 관련 문서 수
	```
	- 정답 근거 문서가 상위 K개 검색 결과 안에서 얼마나 잘 포함됐는지를 나타내는 지표
	- 필요한 근거 문서를 못 가져오면 LLM은 그 문서를 기반으로 답할 수 없기에 중요한 성능 지표임
- **MRR(Mean Reciprocal Rank)**
	```
	RR = 1 / 첫 번째 관련 문서의 순위
	*첫 번째 관련 문서의 순위 : 평가 전 각 쿼리에 대해 '관련 문서'를 사전에 정의해 둔 문서들 중 하나가, 실제 검색 결과 목록에서 가장 먼저 등장한 위치
	
	- 1번째에 나오면 RR = 1/1 = 1.0
	- 2번째에 나오면 RR = 1/2 = 0.5
	- 5번째에 나오면 RR = 1/5 = 0.2
	
	MRR = 모든 질문의 RR 평균
	    = (1/1 + 1/2 + 1/5) / 3
	    = 0.566...
	```
	- 첫 번째 관련 문서가 검색 결과의 몇 번째에 나오는가를 평가하는 지표
	- LLM이 보통 검색 결과 앞쪽 문서에 더 큰 영향을 받기 때문에 RAG에서 MRR이 중요함
	- **전체적인 흐름**
		1. 평가용 질문 쿼리를 여러 개 준비하고, 각 쿼리마다 관련 문서를 정의함
		2. 검색기를 돌려서 결과 순위를 얻음
		3. 각 쿼리 결과에서 사전에 정의한 관련 문서가 몇 번째에 나왔는지 평가함
		4. 그 순위를 그대로 평균내지 않고, `1 / 순위`로 바꾼 뒤 평균냄
- **Precision@K**
	```
	Precision@K = 상위 K개 안의 관련 문서 수 / K
	```
	- 상위 K개 검색 결과 중에서 실제로 관련 있는 문서의 비율을 나타내는 지표
	- 상위 결과에 관련 없는 문서가 섞이면 LLM은 엉뚱한 근거를 읽게 되므로, 해당 지표는 노이즈 관리에 가까움
	- 즉, 해당 지표의 수치가 높다는 것은 검색 결과가 "깨끗하다"는 뜻임
