# AWS Transform을 통한 놀유니버스의 .NET 현대화
Presenter. 지윤성 실장, 놀유니버스

### 현대화 배경

- 아키텍처는 실패해서 바꾸는 게 아니라, 성공했기 때문에 바꿔야 함
	- 기존 레거시 시스템의 검증된 안정성은 아키텍처적 변화에 대한 저항을 만듦
	- 성장의 버팀목이 족쇄가 되는 순간 변화는 선택이 아니라 필수가 됨
- 현대화를 차단하는 3대 고착 요인
	- 심화되는 기술 부채 : Classic ASP/COM+ 기반의 MS 종속 아키텍처로 기술 부채 가중
	- 예측 불가능한 글로벌 폭증 : 국내/글로벌 트래픽 급증으로 기존 인프라 한계 도달
	- 종속적 구조의 병목 : Windows·IIS 종속 구조로 트래픽 대응 한계
- 왜 아키텍처 부분 개선으로는 해결이 불가능한가?
	- 기술 지원이 종료된 .NET 프레임워크를 사용하고 있었기에 부분 개선으로 진행한다면 기술 부채가 오히려 증가
- 현대화는 핵심 비즈니스 중단에 대한 두려움, 큰 변경 리소스, 실패 시 비즈니스 영향으로 인해 미루게 됨
	- 이러한 딜레마에 의해 '잘 돌아가면 건드리지 말자'라는 결론으로 귀결되버림
- 아키텍처 문제의 본질
	- 의존성 사슬 : 하나의 모듈을 수정하면 다른 모듈과 라이브러리까지 연쇄적으로 영향을 받음
	- 실행환경 종속 : 비즈니스 로직이 특정 프레임워크와 실행 방식에 강하게 결합됨
	- 플랫폼 고착 : 특정 OS나 인프라 환경에 고착되어 클라우드 네이티브 전환을 막는 족쇄

### 현대화 관점의 범용 코딩 에이전트의 구조적 한계

1. **대규모 컨텍스트의 벽**
	- 수백만 라인의 코드는 에이전트의 컨텍스트 한계를 넘어서는 경우가 많음
	- 에이전트는 현재 편집 중인 A 파일만 보고, 해당 파일을 의존하는 다른 파일들은 보지 못함
2. **종속성 그래프 해석 불가**
	- 종속성 그래프를 통한 빌드 순서 결정이 필요했지만, 에이전트는 이를 참고하지 않음
3. **오케스트레이션의 부재**
	- 마이그레이션을 위해서는 여러 과정을 통해 세부적인 계획이 필요함 (즉, 오케스트레이션이 필요함)
	- 대량 파일의 변환을 전체적으로 조율하는 것은 범용 에이전트의 설계 범위 밖의 사항임

### AWS Transform

- 에이전트를 이용한 현대화 방식인 에이전틱 현대화를 AWS 서비스로 구현한 방식으로 아래 기능 제공
	- 특화 에이전트 : Windows, Mainframe, VMWare 현대화 전문 에이전트 제공
	- 공유 워크스페이스 및 경험 : 웹 기반의 워크스페이스에서 전체 현황을 통합 관리
- AWS Transform for .NET은 .NET 현대화에 특화된 에이전틱 AI 시스템
- '분석 -> 분할 -> 계획 -> 실행 단계'로 AWS Transform 오케스트레이션 진행
- 전문 도구(AWS Transform이 구조 변환) + 범용 도구(Kiro CLI가 후속 안정화)
